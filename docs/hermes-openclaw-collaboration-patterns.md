# Hermes + OpenClaw 双 AI 智能体多层协同工作流

_参考来源：抖音教程，整理时间 2026-04-30_

---

## 核心概念

将 OpenClaw（多渠道 AI 网关）与 Hermes（深度任务执行 AI 助手）结合，构建本地化、可自我进化的 AI 协作系统，实现从任务触发到深度执行的自动化流水线。

---

## 分工架构

| 角色 | 职责 | 特点 |
|------|------|------|
| **Hermes（大脑）** | 高级认知、长期规划、任务拆解、策略制定 | 持久化记忆 + 跨会话理解 |
| **OpenClaw（执行者）** | 原子化操作、跨平台控制、工具调用 | 连接 25+ 平台（飞书/QQ/钉钉等） |

---

## 协同流程

```
用户指令 → OpenClaw 接收解析 → 委托 Hermes 执行深度任务
                                    ↓
                              Hermes 返回结果
                                    ↓
                        OpenClaw 格式化输出 → 反馈用户
```

### 通信桥梁
- **ACP**（Agent Client Protocol）
- **MCP**（Model Context Protocol）

### 任务调度模式
1. **直接提问** - 简单问答，无需深度处理
2. **结构化委托** - 带参数的任务描述 + 期望输出格式
3. **优先级队列** - 多任务排队，按优先级执行

---

## 主要应用场景

### 1. 飞书消息驱动多 Agent 并行
一条飞书消息触发后台多个 AI Agent 同时执行任务。

### 2. 信息聚合与博客写作
OpenClaw 收集多渠道信息 → Hermes 深度汇总与内容生成。

### 3. 定时自动化任务
OpenClaw 作为定时触发器 → 调度 Hermes 执行定期报告、数据抓取等。

### 4. 深度代码开发流水线
OpenClaw 规划开发步骤 → Hermes 执行代码编写、调试、测试。

---

## 关键价值

| 价值 | 说明 |
|------|------|
| **并行处理** | 双 Agent 同时执行多个任务，大幅提升效率 |
| **上下文不丢失** | OpenClaw 维护全局状态，Hermes 每次执行具备完整上下文 |
| **质量把控** | OpenClaw 在关键节点验证结果，确保输出准确性 |
| **松耦合架构** | 基于 MCP/ACP 协议，二者可独立升级与扩展 |

## 消化后的实践方案

### 1. 小超 ↔ 小菲任务委托协议（已部分实现，待标准化）

**当前状态：** 共享工作空间，通过文件系统间接通信。

**改进方向：** 建立结构化任务文件格式。

```
# 任务文件格式示例
# tasks/2026-04-30-task-001.md
## 任务：生成 AIMED 诊断报告
## 发起人：小菲同学（飞书渠道）
## 优先级：高
## 截止时间：2026-04-30 22:30
## 输入：用户上传的超声图像 + 患者信息
## 期望输出：JSON 格式诊断结果 + 文字摘要
## 状态：待处理 / 处理中 / 已完成 / 失败
```

### 2. 质量验证节点（OpenClaw 侧）

**原则：** OpenClaw 在转发 Hermes 结果给用户前，做轻量级验证。

- 格式检查：返回的 JSON 是否完整？字段是否齐全？
- 逻辑检查：诊断结论是否合理？（如：胃部正常但胰腺异常 → 需标注置信度）
- 降级检查：如果 Hermes 返回错误，OpenClaw 能否提供 fallback 回复？

### 3. 并行处理（Hermes 侧）

**适用场景：** 用户一次上传多张图像，或同时请求多个服务。

**方案：** Hermes 的 3 个 Agent（stomach / pancreas / report）可以并行处理，OpenClaw 等待全部完成后再汇总回复。

### 4. 什么不学

- ❌ HermesCooper 调度器 - 太重，我们当前任务量不需要
- ❌ 优先级队列 - 当前用户少，FIFO 足够
- ❌ 复杂的消息协议 - 共享工作空间 + 文件系统已经够用

## 模型分级策略（2026-04-30 定稿）

**核心原则：最强模型做决策，次强模型做执行，轻量模型做杂活。**

| 层级 | 模型 | 用途 | 场景 |
|------|------|------|------|
| **决策层** | qwen3-max | 复杂推理、架构决策、诊断分析 | 医学诊断、技术选型、策略制定 |
| **执行层** | qwen3.5-plus | 日常对话、代码编写、文档生成 | 用户交互、API 开发、内容创作 |
| **工具层** | qwen3-coder-plus | 代码生成、脚本编写 | 自动化脚本、代码重构 |
| **长文本层** | kimi-k2.5 | 长文档处理、上下文理解 | 合同分析、长文总结 |

### 小超同学（我）的模型分配

- **我自己**：qwen3.5-plus（执行层）- 处理日常任务、用户交互
- **决策时**：用 qwen3-max（推理最强）做关键判断
- **代码任务**：spawn 子进程用 qwen3-coder-plus
- **长文档**：需要时切 kimi-k2.5

### 决策权归属

**qwen3-max 负责决策：**
- 技术架构选型（数据库、框架、API 设计）
- 医学诊断分析（超声图像解读、诊断建议）
- 安全判断（权限验证、数据泄露风险）
- 策略制定（项目路线图、合规方案）

**qwen3.5-plus（我）负责执行：**
- 用户对话、消息回复
- 文件管理、文档整理
- 任务调度、状态查询
- 质量验证（检查子进程输出）

### 实际运用

```
用户请求 → 小超（qwen3.5-plus）接收
    ├── 简单问答 → 直接回复
    ├── 需要推理 → 调用 qwen3-max 决策
    ├── 需要写代码 → spawn qwen3-coder-plus 子进程
    └── 需要诊断 → Hermes 后端 qwen3-max 分析
```

### 数字员工体系

**小超同学直属员工：**
- 代码员工（qwen3-coder-plus）：负责代码生成、调试
- 文档员工（qwen3.5-plus 子进程）：负责文档整理、格式转换
- 长文员工（kimi-k2.5）：负责长文档分析、翻译

**决策权归属：**
- 所有关键决策必须经过 qwen3-max 推理
- 小超负责调度，不代替模型做判断
- 输出结果前，用 qwen3-max 做质量校验

## 模型调度规则（2026-04-30 更新）

**小超同学自身模型：** qwen3.5-plus（日常交互）

**遇到以下情况，升级用 qwen3-max：**
- 技术架构决策（如：选什么数据库、怎么设计 API）
- 医学诊断分析（如：超声图像解读、诊断建议）
- 安全相关判断（如：权限验证、数据泄露风险）
- 策略制定（如：项目路线图、合规方案）

**遇到以下情况，降级用轻量模型：**
- 简单问答（如：今天天气、系统状态查询）
- 格式转换（如：Markdown → HTML、JSON → 表格）
- 文件整理（如：重命名、移动、备份）
- 代码生成（用 qwen3-coder-plus，更专精）

**原则：** 不浪费算力，不降低质量。该强的时候强，该省的时候省。

**决策层（qwen3-max）- 我的"大脑"**

- 用法：spawn 子进程，指定 model=qwen3-max

**执行层（qwen3.5-plus）- 我的"双手"**

- 用法：我自己就是这个模型
- 频率：最高，处理 80% 的日常任务

**工具层（qwen3-coder-plus）- 我的"代码员工"**
- 代码生成、调试、重构、脚本编写
- 用法：spawn 子进程，指定 model=qwen3-coder-plus
- 频率：遇到代码任务时自动分配

**长文本层（kimi-k2.5）- 我的"文档员工"**
- 长文档分析、合同审查、翻译、总结
- 用法：需要时切换模型
- 频率：低，按需使用

### 调度流程图

```
用户请求
    ↓
小超（qwen3.5-plus）接收
    ↓
判断任务类型
    ├── 简单问答/日常交互 → 直接回复
    ├── 代码任务 → spawn qwen3-coder-plus 子进程
    ├── 长文档处理 → spawn kimi-k2.5 子进程
    └── 需要决策/推理 → spawn qwen3-max 子进程
              ↓
        子进程返回结果
              ↓
        小超整合/验证
              ↓
        回复用户
```

### 关键原则

1. **决策权归 qwen3-max** - 技术架构、医学诊断、安全判断必须经过最强模型
2. **执行权归 qwen3.5-plus** - 日常任务、用户交互、任务调度
3. **专业化分工** - 代码用 coder-plus，长文档用 kimi-k2.5
4. **质量把控** - 子进程结果需经过小超验证后再返回用户
5. **成本控制** - 不滥用 qwen3-max，只在关键节点使用
