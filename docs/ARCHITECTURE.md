## 🏗️ 分层设计

### 1. 客户端层 (Client Layer)

**职责：** 用户交互和数据展示

**组成：**
- Web 浏览器（Swagger UI、自定义前端）
- 移动 APP（iOS/Android）
- 第三方系统（医院 HIS/PACS）
- 超声设备（直接对接）

**通信协议：** HTTP/REST API

---

### 2. API 网关层 (API Gateway Layer)

**职责：** 请求入口、安全验证、流量控制

**组件：**
- **FastAPI** - Web 框架
- **CORS 中间件** - 跨域资源共享
- **全局异常处理** - 统一错误响应
- **请求验证** - Pydantic 模型验证

**关键文件：**
```
main.py              # 应用入口
middleware/          # 中间件模块
config.py            # 配置管理
```

---

### 3. 业务路由层 (Router Layer)

**职责：** API 端点定义、请求分发、响应格式化

**路由模块：**
| 路由 | 前缀 | 功能 |
|------|------|------|
| `conductor` | `/api/conductor` | 任务调度 |
| `stomach` | `/api/stomach` | 胃诊断 |
| `pancreas` | `/api/pancreas` | 胰腺诊断 |
| `report` | `/api/report` | 报告生成 |
| `v1/*` | `/api/v1/*` | 版本化 API |

**关键特性：**
- 版本化管理（v1, v2...）
- 请求/响应模型定义
- 输入验证和错误处理

---

### 4. Agent 层 (Agent Layer)

**职责：** 核心业务逻辑、AI 推理、结果整合

#### 4.1 Conductor Agent（总指挥）

**职责：**
- 任务接收和解析
- Agent 注册和发现
- 任务分发和调度
- 结果整合和返回

**核心方法：**
```python
register_agent(name, agent)     # 注册 Agent
dispatch_task(organs, images)   # 分发任务
get_task_status(task_id)        # 查询状态
complete_task(task_id, result)  # 完成任务
```

#### 4.2 Stomach Agent（胃诊断）

**职责：**
- 胃超声影像分析
- 病变识别和分类
- 置信度评估
- 诊断建议生成

**处理流程：**
```
影像输入 → 预处理 → AI 推理 → 结果后处理 → 诊断输出
```

#### 4.3 Pancreas Agent（胰腺诊断）

**职责：**
- 胰腺超声影像分析
- 病灶检测（囊肿、肿瘤等）
- 风险分层
- 随访建议

#### 4.4 Report Agent（报告生成）

**职责：**
- 结构化报告生成
- 多语言支持（6 种）
- PDF 导出
- 模板管理

---

### 5. 服务层 (Service Layer)

**职责：** 外部服务集成、工具调用

#### 5.1 AI 模型服务

**当前：** 阿里云百炼（DashScope）
- Qwen-Plus 大模型
- 医学知识 RAG
- 诊断推理

**未来：** 本地部署模型
- 7B 蒸馏模型（器官级）
- 72B 主模型（复杂病例）

#### 5.2 影像处理服务

**工具栈：**
- **PyDICOM** - DICOM 文件解析
- **SimpleITK** - 医学图像处理
- **Pillow** - 通用图像操作
- **NumPy** - 数值计算

**功能：**
- 格式转换（DICOM → PNG）
- 图像增强
- 尺寸标准化
- 质量评估

#### 5.3 报告生成服务

**工具栈：**
- **ReportLab** - PDF 生成
- **Jinja2** - 模板渲染

**功能：**
- 结构化报告
- 多语言模板
- PDF 导出
- 数字签名（未来）

---

### 6. 数据层 (Data Layer)

**职责：** 数据持久化、文件存储、日志记录

#### 6.1 数据库

**当前：** SQLite（开发/测试）
- 轻量级
- 零配置
- 单文件

**生产环境：** PostgreSQL
- 高并发
- 数据备份
- 事务支持

**数据模型：**
- 患者信息
- 诊断记录
- 报告元数据
- 用户账户

#### 6.2 文件存储

**存储内容：**
- 超声影像（原始 + 处理后）
- 生成的 PDF 报告
- 模型权重文件

**目录结构：**
```
data/
├── samples/     # 测试样本
├── reports/     # 生成的报告
└── uploads/     # 用户上传
```

#### 6.3 日志系统

**工具：** Loguru

**日志级别：**
- DEBUG - 调试信息
- INFO - 一般信息
- WARNING - 警告
- ERROR - 错误
- CRITICAL - 严重错误

**日志内容：**
- 请求日志
- 诊断日志
- 错误堆栈
- 性能指标

## 🔄 数据流

### 典型诊断流程

```
1. 用户上传影像
   ↓
2. API 网关接收请求
   ↓
3. 路由验证和分发
   ↓
4. Agent 执行诊断
   ↓
5. AI 模型推理
   ↓
6. 结果后处理
   ↓
7. 返回诊断结果
   ↓
8. （可选）生成报告
```

### 任务调度流程

```
1. Conductor 接收多器官诊断请求
   ↓
2. 解析请求，验证器官类型
   ↓
3. 并行分发到各器官 Agent
   ↓
4. 等待所有 Agent 完成
   ↓
5. 整合结果
   ↓
6. 返回综合诊断报告
```

## 🔐 安全架构

### 认证授权

**当前：** 无（开发环境）

**计划：** JWT Token
- 用户登录获取 Token
- 请求携带 Token 验证
- 24 小时过期刷新

### 数据安全

- 患者数据脱敏
- API Key 环境变量隔离
- 敏感信息不记录日志
- HTTPS 加密传输（生产）

### 访问控制

- CORS 限制来源
- 请求频率限制（未来）
- IP 白名单（未来）

## 📊 性能设计

### 并发处理

- **异步 IO：** FastAPI + asyncio
- **多进程：** Uvicorn workers
- **并行诊断：** 多 Agent 同时执行

### 缓存策略

- **配置缓存：** `@lru_cache()` 配置单例
- **结果缓存：** 相同影像复用结果（未来）
- **模型缓存：** 模型预加载到内存

### 优化方向

- 模型量化（INT8/INT4）
- 模型蒸馏（72B → 7B）
- MOE 专家路由
- GPU 加速推理
